# LingoLens Codebase Audit — 2026-06-12

## 1. Executive Summary

Overall health score: 6.5/10. Production readiness: Near ready for a controlled MVP, not ready for public production with real learner/admin data. The codebase has a coherent Next.js/Prisma architecture, a multilingual-ready data model, server-side OpenAI usage, Zod validation, sanitized Markdown rendering, and strong visual direction from the Stitch export. The biggest risks are production data/change control (`Dockerfile:22`, missing Prisma migrations), generation reliability (`app/admin/actions.ts:54`), incomplete URL/media hardening (`lib/media.ts:62`, `lib/validators.ts:9`), and no automated tests. Best quick win: fix the admin edit locale preservation bug, dashboard no-op search, missing CSS tokens, publish guards, RSS image metadata, and transitive PostCSS advisory.

- Biggest security risk: arbitrary admin-provided image URLs are fetched server-side without protocol/private-network/content-size controls in `lib/media.ts:62`.
- Biggest UX issue: homepage and cards do not fully match the required Stitch flow; `app/page.tsx:34` omits the how-it-works section and `components/ArticleCard.tsx:24` omits level chips.
- Biggest technical debt issue: runtime `pnpm prisma:push && pnpm seed` in `Dockerfile:22` with no `prisma/migrations/` directory.
- Best quick win: safe validation/UI guard batch listed in section 19.

## 2. App Understanding

LingoLens is a Next.js 16 App Router app for manually entered source texts that generate and publish locale + reading-level adaptations. The MVP target is `es-419`, but Prisma models correctly use `Locale`, `ReadingLevel`, `LocaleLevelProfile`, `ContentItem`, `Adaptation`, `FactBank`, `GenerationJob`, `MediaAsset`, `ContentRightsRecord`, `PromptTemplate`, `AdaptationRevision`, and `RssFeedConfig` instead of language-specific columns (`prisma/schema.prisma:74`, `prisma/schema.prisma:226`).

Tech stack: Next.js App Router, React, TypeScript strict mode, Prisma/PostgreSQL, Zod, OpenAI Node SDK, React Markdown with `rehype-sanitize`, Sharp, Docker Compose. Routing structure: public `/`, `/articles`, `/feeds`, `/feeds/[locale]/[level].xml`, `/read/[locale]/[level]/[slug]`, `/media/[...path]`; admin `/admin`, `/admin/login`, `/admin/content`, `/admin/content/new`, `/admin/content/[id]/edit`, `/admin/content/[id]/review`.

Main flows: readers browse published adaptations and switch levels; admins log in, create/edit source content, generate adaptations, review/edit generated versions, publish/archive, and view generation failures. Auth uses a signed `ll_admin` cookie (`lib/auth.ts:7`) backed by database `AdminUser` records (`prisma/schema.prisma:308`), with server-side `requireAdmin()` checks on admin pages/actions. OpenAI generation is server-only and falls back to deterministic mock generation when `OPENAI_API_KEY` is missing or placeholder (`lib/generation.ts:74`). Deployment assumes Docker Compose with Postgres and `DATA_DIR` persistent bind mounts (`docker-compose.yml:16`, `docker-compose.yml:38`), but the app container applies schema with `prisma db push` at startup (`Dockerfile:22`).

Testing setup: no test files found and no `test` or `format:check` scripts in `package.json:6`. Validation commands were run and are recorded in section 23.

## 3. Top 10 Findings

1. P1 / High / Deployment and data safety — `Dockerfile:22`, `prisma/` — production startup runs `prisma:push` and `seed` without migrations. Why it matters: schema drift can be applied without review and seed behavior runs on every boot. Recommended fix: add migrations and use `prisma migrate deploy`; keep seed/admin bootstrap explicit or tightly gated.
2. P1 / High / Reliability — `app/admin/actions.ts:54`, `lib/generation.ts:334` — generation runs as detached background work from a server action. Why it matters: process restarts or serverless-like request lifecycle can strand jobs/content in `generating`. Recommended fix: persist queued jobs and process them in an explicit worker or supervised internal job runner.
3. P1 / High / Security/privacy — `lib/media.ts:62`, `lib/validators.ts:12` — admin-supplied image URLs were accepted without scheme checks and are still fetched server-side without private IP, size, MIME, timeout, or redirect limits. Why it matters: SSRF/cost and memory abuse risk. Recommended fix: the safe-fix pass added `http/https` validation; finish DNS/private-network/size/MIME hardening.
4. P1 / Medium / Correctness — `app/admin/actions.ts:327` — `Publish All` could mark a content item published even if it had zero publishable adaptations. Why it matters: admin status becomes false and RSS/public listings remain empty. Recommended fix: implemented server-side count guard and hidden empty-state button.
5. P1 / Medium / Rights workflow correctness — `app/admin/actions.ts:282` — `REQUIRE_RIGHTS_APPROVAL_TO_PUBLISH=true` checked text rights only, not image rights. Why it matters: image rights could remain `unreviewed` while publishing was allowed. Recommended fix: implemented image-rights gate.
6. P1 / Medium / Security dependency — `pnpm audit --audit-level low` — PostCSS `<8.5.10` advisory via `next > postcss@8.4.31`. Why it matters: dependency audit failed and the advisory is XSS-related in CSS stringification. Recommended fix: implemented pnpm workspace override to `postcss@8.5.15`; audit now passes.
7. P1 / Medium / Admin UX correctness — `app/admin/page.tsx:139` — dashboard search/filter controls did not submit anywhere. Why it matters: visible controls that do nothing erode trust and slow editors. Recommended fix: implemented GET form to `/admin/content?q=...`.
8. P2 / Medium / Correctness — `components/ContentForm.tsx:36`, `app/admin/content/[id]/edit/page.tsx:13` — edit form could not preserve the current source locale because it received `sourceLocaleId` while options post `bcp47Tag`. Why it matters: future non-English source content could silently save with the first locale. Recommended fix: implemented `sourceLocale` include and BCP 47 default.
9. P2 / Medium / RSS correctness — `app/feeds/[locale]/[level].xml/route.ts:40` — feed items lacked article image metadata and the channel lacked a logo. Why it matters: RSS readers cannot display the LingoLens brand or article art. Recommended fix: implemented channel `<image>` plus Media RSS item image tags and local-image enclosures.
10. P2 / Medium / Performance/testing/tooling — `prisma/schema.prisma`, `package.json:6` — no secondary query indexes and no test/format scripts. Why it matters: scale and regression risks compound. Recommended fix: add indexes plus Vitest/Playwright coverage in follow-up PRs.

- Recommended fix: Generate an initial Prisma migration, change container startup to `pnpm prisma:deploy && pnpm bootstrap-admin && pnpm start`, and make sample seed opt-in for non-production or first-run setup.
- Effort: Medium
- Status: Recommended

### [P1] Move Generation Out Of Detached Server Action Work

- Category: Reliability / Architecture
- Severity: High
- Affected files/functions: `app/admin/actions.ts:54`, `lib/generation.ts:334`, `GenerationJob`
- Problem: `queueGeneration()` sets the content item to `generating`, then starts `void Promise.allSettled(...)` after the request returns.
- Why it matters: a restart, crash, deploy, or serverless-like lifecycle can terminate work with no retry loop. Jobs can remain incomplete and content status can mislead admins.
- Recommended fix: Persist queued jobs, add a bounded worker command or internal polling worker for Docker, mark stale running jobs failed/retryable, and expose retry controls.
- Effort: Large
- Status: Recommended
- Fix prompt: In LingoLens, redesign generation queuing so `app/admin/actions.ts` only creates `GenerationJob` rows and redirects. Add a worker entry point that claims queued jobs, calls the existing `generateAdaptations` and image optimization logic, records status/timestamps/errors, and safely handles retries/stale running jobs. Keep OpenAI keys server-side. Update admin dashboard failure display and validate with unit tests for queued/succeeded/failed job transitions plus `pnpm build`.

### [P1] Harden Remote Image Fetching

- Category: Security / Privacy / Reliability
- Severity: High
- Affected files/functions: `lib/media.ts:62`, `lib/validators.ts:12`, `components/ContentForm.tsx:83`
- Problem: Any syntactically valid URL can be saved as a header image and fetched server-side without request limits.
- Why it matters: an admin mistake or compromised admin session can hit private services, metadata endpoints, very large files, or non-image responses. Logs include the source URL (`lib/media.ts:57`, `lib/media.ts:111`).
- Recommended fix: Restrict URL schemes to `http`/`https`, block localhost/private/link-local IPs after DNS resolution, cap content length and bytes read, require image MIME, set timeout, and limit redirects.
- Effort: Medium
- Status: Partially implemented
- Fix prompt: In LingoLens, harden header image ingestion. Update `lib/validators.ts` so `sourceUrl` and `headerImageUrl` only allow `http:` and `https:` URLs. Update `lib/media.ts` to reject private/localhost/link-local/metadata destinations after DNS lookup, use `AbortController` timeouts, reject non-image content types, cap downloaded bytes according to `MAX_UPLOAD_MB`, and limit redirects. Keep errors logged without leaking secrets. Validate with unit tests for allowed HTTPS image URLs, non-http URLs, localhost/private IP URLs, huge responses, and non-image MIME responses, then run `pnpm lint` and `pnpm build`.

## 5. Security Audit

### Critical

No confirmed Critical/P0 issue found.

- Location: `app/admin/actions.ts:282`
  - Risk: rights gate only checks text rights and ignores image rights.
  - Exploit scenario: with `REQUIRE_RIGHTS_APPROVAL_TO_PUBLISH=true`, content with unreviewed image rights can still publish.

  - Immediate action needed: Implemented in `app/admin/actions.ts`.

  - Risk: no login rate limiting or lockout.
